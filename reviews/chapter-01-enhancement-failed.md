## Conceptual Foundation

Authentication and authorization are foundational security controls that protect applications from unauthorized access. While often discussed together, they serve distinct purposes and must be implemented with equal rigor.

**Authentication** is the process of verifying that a user is who they claim to be. It answers the question: "Are you really Alice?" Authentication establishes identity through one or more factors: something you know (password), something you have (hardware token, phone), or something you are (biometric). The strength of authentication depends on the factors used and how securely they are managed.

**Authorization** is the process of determining what an authenticated user is permitted to do. It answers the question: "What resources can Alice access, and what actions can she perform?" Authorization enforces access control policies that define which users can perform which operations on which resources.

The relationship between these concepts is sequential but inseparable. Authentication must occur before authorization can be meaningfully applied. A system cannot determine what an unauthenticated user should access. Conversely, authentication alone is insufficient—a successfully authenticated user still requires authorization checks to prevent them from accessing resources beyond their privilege level. This separation of concerns enables independent evolution of identity verification and access control policies.

### Authentication Factors

Authentication strength is determined by the number and type of factors used:

**Single-Factor Authentication (Something You Know):** Typically a password or PIN. This factor is vulnerable to phishing, keylogging, password reuse, and brute force attacks. Passwords remain the most common authentication mechanism but should not be the only factor for sensitive operations. The OWASP Top 10 consistently identifies weak authentication as a critical vulnerability class. Single-factor systems are particularly vulnerable to credential stuffing attacks, where attackers use credentials compromised in other breaches to gain access.

**Two-Factor Authentication (2FA):** Combines two different factors, such as a password plus a time-based one-time password (TOTP) generated by an authenticator app, or a password plus a push notification to a registered device. This significantly increases security because compromising one factor does not grant access. However, not all 2FA methods are equally secure: SMS-based 2FA is vulnerable to SIM swapping and interception, while app-based TOTP and hardware tokens provide stronger protection. Push notification-based 2FA can be vulnerable to notification fatigue attacks, where attackers send repeated notifications hoping the user will approve one by mistake.

**Multi-Factor Authentication (MFA):** Uses three or more factors. For example, password + TOTP + biometric. MFA provides the strongest authentication assurance but introduces complexity and potential usability challenges. The most secure implementations use phishing-resistant factors such as FIDO2/WebAuthn hardware keys, which cryptographically bind authentication to the legitimate service and cannot be phished. Organizations should prioritize phishing-resistant MFA for high-value accounts (administrators, finance, executives).

### Authorization Models

Authorization is implemented through several conceptual models:

**Role-Based Access Control (RBAC):** Users are assigned roles (e.g., "Administrator," "Editor," "Viewer"), and permissions are granted to roles rather than individual users. This simplifies management at scale but can become inflexible if roles are not designed carefully. A common failure mode is role creep, where roles accumulate permissions over time as new features are added, violating the principle of least privilege. RBAC works well for applications with clear organizational hierarchies but struggles with complex, context-dependent access requirements.

**Attribute-Based Access Control (ABAC):** Access decisions are based on attributes of the user, resource, action, and environment. For example, "Allow access to patient records if the user is a doctor AND the patient has consented AND the access is during business hours AND the user is on the corporate network." ABAC is more flexible and better suited to complex authorization requirements but requires careful policy design and evaluation. ABAC policies can become difficult to audit and debug if not properly documented. Performance can be a concern if policy evaluation is not optimized.

**Access Control Lists (ACLs):** Explicit lists define which users or groups can perform which actions on specific resources. ACLs are granular but do not scale well as the number of users and resources grows. ACLs are useful for fine-grained control over specific resources but should be combined with role-based controls for broader permission management.

**Principle of Least Privilege (PoLP):** Users should have only the minimum permissions necessary to perform their job function. This principle applies across all authorization models and is essential for reducing the impact of compromised accounts. PoLP is not a one-time configuration but an ongoing practice: permissions should be regularly audited and revoked when no longer needed. Temporary elevation of privileges (e.g., for administrative tasks) should be time-bound and logged.

## Architecture Perspective

Authentication and authorization must be designed as integral components of application architecture, not as afterthoughts bolted onto existing systems. Architectural decisions made early in development significantly impact security, scalability, and maintainability.

### Authentication Architecture Patterns

**Centralized Authentication Service:** A dedicated authentication service (such as an identity provider) handles all authentication logic. Applications delegate authentication to this service and trust the tokens or assertions it issues. This pattern simplifies application code, enables single sign-on (SSO), and centralizes security policy enforcement. Advantages include consistent password policies, centralized MFA management, and simplified credential rotation. Disadvantages include increased complexity, potential single point of failure, and latency if the authentication service is unavailable. Organizations should implement fallback mechanisms and high availability for centralized authentication services.

```
User → Application → Identity Provider → User Database
                          ↓
                    Validate Credentials
                          ↓
                    Issue Token/Assertion
                          ↓
                    Return to Application
```

**Distributed Authentication:** Each application maintains its own user database and authentication logic. This pattern is simpler for small systems but creates security and operational challenges: password policies may be inconsistent, credential compromise in one application may not be detected in others, and users must manage multiple credentials. Distributed authentication should be avoided in favor of centralized approaches as systems grow. If distributed authentication is necessary, implement strong synchronization mechanisms and consistent security policies across all applications.

**Federated Authentication:** Multiple organizations or systems trust a common identity provider. Users authenticate once and gain access to resources across multiple systems. This pattern is common in enterprise environments and B2B scenarios. Federated authentication requires careful trust establishment and validation of identity assertions from external providers. Organizations should implement attribute mapping to translate external identity attributes to local authorization decisions.

### Authorization Architecture Patterns

**Authorization at the API Gateway:** A gateway or reverse proxy intercepts requests and enforces authorization before they reach backend services. This pattern centralizes policy enforcement but can become a bottleneck and may not be suitable for fine-grained, resource-specific authorization decisions. Gateway-based authorization works well for coarse-grained access control (e.g., "only authenticated users can access this API") but struggles with resource-level decisions that require knowledge of the specific resource being accessed. Gateway authorization should be combined with application-level checks for defense in depth.

**Authorization in Application Code:** Each application enforces authorization for the resources and operations it manages. This pattern allows fine-grained control but requires careful implementation in every application and can lead to inconsistencies. Application-level authorization enables resource-specific decisions and context-aware policies but requires developers to implement authorization correctly in every endpoint. Code review and automated testing are essential to catch authorization flaws.

**Authorization Service:** A dedicated service evaluates authorization policies and returns access decisions. Applications query this service before performing sensitive operations. This pattern centralizes policy management and enables complex authorization logic but introduces latency and requires careful caching strategies. Authorization services should be designed for high availability and low latency. Caching of authorization decisions must be carefully managed to balance performance with the need to revoke access quickly.

### Token-Based Authentication

Modern applications frequently use tokens (such as JSON Web Tokens or JWTs) instead of session cookies for authentication. Tokens are self-contained, can be validated without querying a database, and work well in distributed and mobile architectures. However, token-based authentication introduces new security considerations around token storage, expiration, and revocation.

A typical token-based flow:

1. User submits credentials to the authentication service
2. Authentication service validates credentials and issues a signed token containing user identity and claims
3. User includes the token in subsequent requests (typically in the Authorization header)
4. Application validates the token signature and extracts user identity and claims
5. Application uses the token claims to make authorization decisions

Token-based authentication requires careful attention to token expiration, refresh token management, and secure storage of signing keys. A critical failure mode is using tokens with excessive expiration times (days or weeks), which increases the window of compromise if a token is stolen. Another common mistake is storing sensitive information in token claims that should not be exposed to the client.

## AppSec Lens

From an application security perspective, authentication and authorization are high-risk areas where implementation flaws directly enable unauthorized access. These controls are frequently targeted by attackers and are among the most commonly exploited vulnerability classes.

### Authentication Risks

**Weak Password Policies:** Applications that allow short passwords, do not enforce complexity requirements, or permit common passwords are vulnerable to brute force and dictionary attacks. An attacker with a list of common passwords can compromise accounts in minutes. NIST SP 800-63B recommends prioritizing password length (minimum 12 characters) and checking against common password lists over complexity rules. Overly complex requirements (uppercase, lowercase, numbers, symbols) encourage users to choose predictable patterns like "P@ssw0rd!" which are easily guessed.

**Credential Storage Vulnerabilities:** Passwords stored in plaintext or with weak hashing algorithms (MD5, SHA1) can be compromised if the database is breached. Even with strong hashing, inadequate salt or iteration counts can enable offline brute force attacks. A compromised password database with weak hashing can be cracked in hours or days using modern GPU-accelerated tools. Organizations should use bcrypt, scrypt, or Argon2 with appropriate iteration counts (bcrypt rounds ≥ 12, Argon2 with memory ≥ 64MB).

**Session Management Flaws:** Sessions that do not expire, use predictable identifiers, or are not invalidated on logout enable session hijacking and account takeover. Sessions transmitted over unencrypted connections are vulnerable to interception. Session identifiers should be cryptographically random, at least 128 bits of entropy. Sessions should expire after a period of inactivity (15-30 minutes for sensitive operations, longer for less sensitive). Session invalidation on logout must be enforced on the server side; client-side deletion of cookies is insufficient.

**Insecure Password Recovery:** Password reset mechanisms that do not verify user identity, send passwords via email, or use predictable reset tokens enable account takeover. An attacker who can reset a victim's password gains full account access. Password reset tokens should be time-limited (15-30 minutes), single-use, and cryptographically random. The application should verify user identity before allowing a reset (security questions, email verification, SMS code). Never send passwords via email or display them in the UI.

**Lack of Multi-Factor Authentication:** Applications that rely solely on passwords for authentication are vulnerable to phishing, credential stuffing, and brute force attacks. MFA significantly raises the bar for attackers. Organizations should mandate MFA for all users, with particular emphasis on high-value accounts (administrators, finance, executives). Phishing-resistant MFA (FIDO2/WebAuthn) should be prioritized over SMS or email-based factors.

**Insufficient Authentication Logging:** Applications that do not log authentication events cannot detect account compromise or brute force attacks. Logging should capture successful and failed authentication attempts, including timestamp, username, and source IP. Logs should be protected from tampering and retained for sufficient time to enable forensic analysis. Organizations should implement alerting on suspicious patterns such as multiple failed attempts from the same IP, logins from geographically impossible locations, or logins at unusual times.

**Credential Stuffing and Brute Force:** Attackers use credentials compromised in other breaches to attempt login to applications. Without rate limiting or account lockout, attackers can test thousands of credentials quickly. Organizations should implement progressive delays after failed attempts, account lockout after a threshold of failures, and CAPTCHA challenges. Rate limiting should be applied per username and per IP address to catch both targeted attacks (many attempts on one account) and distributed attacks (few attempts per IP).

### Authorization Risks

**Broken Access Control:** The most common authorization vulnerability is the failure to enforce access control checks. For example, an application might check authorization when displaying a resource but fail to check when the user attempts to modify it. An attacker can bypass the UI and directly request the modification endpoint. Authorization checks must occur on the server, not just in the UI. Every endpoint that accesses or modifies data should verify authorization before proceeding.

**Privilege Escalation:** Vulnerabilities that allow a user to gain higher privileges than intended. For example, a user might modify a parameter in a request to change their role from "User" to "Administrator," or an application might fail to validate that a user has permission to perform an action on behalf of another user. Privilege escalation can be vertical (gaining higher privilege levels) or horizontal (accessing resources of other users at the same level). Both are critical vulnerabilities.

**Insecure Direct Object References (IDOR):** An application exposes object identifiers (such as user IDs or document IDs) in URLs or API parameters without proper authorization checks. An attacker can enumerate identifiers and access resources belonging to other users. IDOR is one of the most common authorization vulnerabilities. Remediation involves verifying authorization before returning any resource. Using indirect references (UUIDs instead of sequential IDs) makes enumeration more difficult but does not prevent IDOR if authorization checks are missing.

**Path Traversal and Directory Traversal:** An application fails to validate file paths, allowing an attacker to access files outside the intended directory. For example, a request for `/documents/../../etc/passwd` might return the system password file. Path traversal vulnerabilities are particularly dangerous in file upload and download functionality. Applications should validate file paths, use allowlists of permitted directories, and avoid constructing paths from user input.

**Horizontal Privilege Escalation:** A user gains access to resources belonging to other users at the same privilege level. For example, a user might access another user's profile or financial records. Horizontal escalation is often enabled by IDOR vulnerabilities. Remediation involves verifying that the authenticated user has permission to access the specific resource being requested.

**Vertical Privilege Escalation:** A user gains access to resources or operations restricted to higher privilege levels. For example, a regular user might access administrative functions. Vertical escalation can result from missing authorization checks, trusting user-supplied role parameters, or flawed authorization logic. Remediation involves verifying the user's actual role from a trusted source (session, token) and checking authorization before performing sensitive operations.

**Insecure Authorization Logic:** Authorization decisions based on user input, client-side checks only, or flawed logic. For example, an application might trust a user-supplied role parameter instead of retrieving the role from a secure session. Authorization logic should never trust user-supplied parameters for security decisions. All authorization decisions should be made on the server, based on the authenticated user's identity and permissions retrieved from a trusted source.

**Inconsistent Authorization Across Interfaces:** Authorization logic that is implemented in the web UI but not in the API, or implemented for some HTTP methods but not others. An attacker can bypass UI-based authorization by directly calling the API or using a different HTTP method. Authorization should be enforced consistently across all interfaces and HTTP methods.

## Developer Lens

Developers must implement authentication and authorization correctly to prevent unauthorized access. This section provides practical guidance for secure implementation, including common pitfalls and how to avoid them.

### Authentication Implementation

**Password Hashing:** Never store passwords in plaintext. Use a modern password hashing algorithm such as bcrypt, scrypt, or Argon2. These algorithms are intentionally slow, making brute force attacks computationally expensive. Use a unique salt for each password (modern algorithms handle this automatically). Iteration counts should be set high enough to make brute force attacks impractical: bcrypt rounds ≥ 12 (taking ~100ms per hash), Argon2 with memory ≥ 64MB and time ≥ 2 iterations.

```python
# Secure password hashing with bcrypt
import bcrypt

# During registration
password = "user_provided_password"
hashed = bcrypt.hashpw(password.encode('utf-8'), bcrypt.gensalt(rounds=12))
# Store hashed in database

# During login
provided_password = "user_provided_password"
if bcrypt.checkpw(provided_password.encode('utf-8'), stored_hashed):
    # Password matches, authenticate user
    pass
else:
    # Password does not match, reject login
    pass
```

**Password Policy:** Enforce minimum length (at least 12 characters) and check against common password lists. NIST SP 800-63B recommends prioritizing length and dictionary checking over complexity rules (uppercase, lowercase, numbers, symbols). Overly complex requirements encourage weak patterns such as predictable substitutions ("P@ssw0rd"). Consider using a password strength meter to guide users toward stronger passwords. Avoid password expiration policies unless required by compliance; they encourage users to choose weak passwords and write them down. Instead, focus on detecting compromised passwords and requiring immediate changes.

**Session Management:** Use secure, randomly generated session identifiers with at least 128 bits of entropy. Set appropriate expiration times (shorter for sensitive operations like financial transactions, longer for less sensitive). Invalidate sessions on logout. Use secure cookie flags: `HttpOnly` (prevents JavaScript access, mitigating XSS-based session theft), `Secure` (HTTPS only, preventing interception), and `SameSite` (prevents CSRF and cross-site session fixation). The `SameSite=Strict` flag is most secure but may break legitimate cross-site navigation; `SameSite=Lax` provides a reasonable balance.

```javascript
// Setting a secure session cookie in Node.js/Express
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,        // HTTPS only
    httpOnly: true,      // No JavaScript access
    sameSite: 'strict',  // CSRF protection and session fixation prevention
    maxAge: 3600000      // 1 hour expiration
  }
}));
```

**Session Storage and Fixation Prevention:** Store sessions in a secure backend (Redis, database) rather than relying on client-side cookies alone. Regenerate session identifiers after successful authentication to prevent session fixation attacks, where an attacker forces a user to use a known session ID. Implement concurrent session limits to detect account takeover: if a user logs in from a new location, invalidate previous sessions or require re-authentication. Track session metadata (IP address, user agent) and alert users to logins from unusual locations.

**Multi-Factor Authentication:** Implement MFA for sensitive operations or user accounts. Support multiple MFA methods (TOTP, push notifications, hardware tokens) to accommodate different user preferences and security requirements. Prioritize phishing-resistant methods such as FIDO2/WebAuthn hardware keys over SMS or email-based factors. SMS-based 2FA is vulnerable to SIM swapping and should be avoided for high-value accounts. Provide backup codes for account recovery if the primary MFA method is unavailable; backup codes should be single-use and stored securely (hashed, like passwords).

**Token-Based Authentication:** If using tokens, ensure they are signed with a strong algorithm. For distributed systems (microservices, APIs), use asymmetric algorithms (RS256 or ES256) where the authentication service signs tokens with a private key and applications validate with the public key. For single-service scenarios, HS256 with a strong shared secret is acceptable, but requires careful key distribution. Include an expiration time (`exp` claim) and validate it on every request. Use short-lived access tokens (15 minutes to 1 hour) and refresh tokens (days to weeks) to limit the window of compromise. Never include sensitive information (passwords, API keys, PII) in token claims.

```javascript
// Validating a JWT token in Node.js
const jwt = require('jsonwebtoken');

function verifyToken(token) {
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      algorithms: ['RS256'],  // Asymmetric signature
      issuer: 'https://auth.example.com',
      audience: 'api.example.com'
    });
    return decoded;
  } catch (error) {
    // Token is invalid, expired, or tampered with
    return null;
  }
}
```

**Token Revocation and Expiration:** Short-lived access tokens (15 minutes to 1 hour) are the most practical approach to token revocation: when a token expires, the user must use a refresh token to obtain a new one. If immediate revocation is required (e.g., after logout or permission change), maintain a revocation list (blacklist) of invalidated tokens, but cache it locally to avoid querying on every request.
