## Conceptual Foundation

Authentication and authorization are foundational security controls that protect applications from unauthorized access. While often discussed together, they serve distinct purposes and must be implemented with equal rigor.

**Authentication** is the process of verifying that a user is who they claim to be. It answers the question: "Are you really Alice?" Authentication establishes identity through one or more factors: something you know (password), something you have (hardware token, phone), or something you are (biometric). The strength of authentication depends on the factors used and how securely they are managed.

**Authorization** is the process of determining what an authenticated user is permitted to do. It answers the question: "What resources can Alice access, and what actions can she perform?" Authorization enforces access control policies that define which users can perform which operations on which resources.

The relationship between these concepts is sequential but inseparable. Authentication must occur before authorization can be meaningfully applied. A system cannot determine what an unauthenticated user should access. Conversely, authentication alone is insufficient—a successfully authenticated user still requires authorization checks to prevent them from accessing resources beyond their privilege level.

### Authentication Factors

Authentication strength is determined by the number and type of factors used:

**Single-Factor Authentication (Something You Know):** Typically a password or PIN. This factor is vulnerable to phishing, keylogging, password reuse, and brute force attacks. Passwords remain the most common authentication mechanism but should not be the only factor for sensitive operations.

**Two-Factor Authentication (2FA):** Combines two different factors, such as a password plus a time-based one-time password (TOTP) generated by an authenticator app, or a password plus a push notification to a registered device. This significantly increases security because compromising one factor does not grant access.

**Multi-Factor Authentication (MFA):** Uses three or more factors. For example, password + TOTP + biometric. MFA provides the strongest authentication assurance but introduces complexity and potential usability challenges.

### Authorization Models

Authorization is implemented through several conceptual models:

**Role-Based Access Control (RBAC):** Users are assigned roles (e.g., "Administrator," "Editor," "Viewer"), and permissions are granted to roles rather than individual users. This simplifies management at scale but can become inflexible if roles are not designed carefully.

**Attribute-Based Access Control (ABAC):** Access decisions are based on attributes of the user, resource, action, and environment. For example, "Allow access to patient records if the user is a doctor AND the patient has consented AND the access is during business hours AND the user is on the corporate network." ABAC is more flexible but requires careful policy design and evaluation.

**Access Control Lists (ACLs):** Explicit lists define which users or groups can perform which actions on specific resources. ACLs are granular but do not scale well as the number of users and resources grows.

**Principle of Least Privilege (PoLP):** Users should have only the minimum permissions necessary to perform their job function. This principle applies across all authorization models and is essential for reducing the impact of compromised accounts.

## Architecture Perspective

Authentication and authorization must be designed as integral components of application architecture, not as afterthoughts bolted onto existing systems.

### Authentication Architecture Patterns

**Centralized Authentication Service:** A dedicated authentication service (such as an identity provider) handles all authentication logic. Applications delegate authentication to this service and trust the tokens or assertions it issues. This pattern simplifies application code, enables single sign-on (SSO), and centralizes security policy enforcement.

```
User → Application → Identity Provider → User Database
                          ↓
                    Validate Credentials
                          ↓
                    Issue Token/Assertion
                          ↓
                    Return to Application
```

**Distributed Authentication:** Each application maintains its own user database and authentication logic. This pattern is simpler for small systems but creates security and operational challenges: password policies may be inconsistent, credential compromise in one application may not be detected in others, and users must manage multiple credentials.

**Federated Authentication:** Multiple organizations or systems trust a common identity provider. Users authenticate once and gain access to resources across multiple systems. This pattern is common in enterprise environments and B2B scenarios.

### Authorization Architecture Patterns

**Authorization at the API Gateway:** A gateway or reverse proxy intercepts requests and enforces authorization before they reach backend services. This pattern centralizes policy enforcement but can become a bottleneck and may not be suitable for fine-grained, resource-specific authorization decisions.

**Authorization in Application Code:** Each application enforces authorization for the resources and operations it manages. This pattern allows fine-grained control but requires careful implementation in every application and can lead to inconsistencies.

**Authorization Service:** A dedicated service evaluates authorization policies and returns access decisions. Applications query this service before performing sensitive operations. This pattern centralizes policy management and enables complex authorization logic but introduces latency and requires careful caching strategies.

### Token-Based Authentication

Modern applications frequently use tokens (such as JSON Web Tokens or JWTs) instead of session cookies for authentication. Tokens are self-contained, can be validated without querying a database, and work well in distributed and mobile architectures.

A typical token-based flow:

1. User submits credentials to the authentication service
2. Authentication service validates credentials and issues a signed token containing user identity and claims
3. User includes the token in subsequent requests (typically in the Authorization header)
4. Application validates the token signature and extracts user identity and claims
5. Application uses the token claims to make authorization decisions

Token-based authentication requires careful attention to token expiration, refresh token management, and secure storage of signing keys.

## AppSec Lens

From an application security perspective, authentication and authorization are high-risk areas where implementation flaws directly enable unauthorized access.

### Authentication Risks

**Weak Password Policies:** Applications that allow short passwords, do not enforce complexity requirements, or permit common passwords are vulnerable to brute force and dictionary attacks. An attacker with a list of common passwords can compromise accounts in minutes.

**Credential Storage Vulnerabilities:** Passwords stored in plaintext or with weak hashing algorithms (MD5, SHA1) can be compromised if the database is breached. Even with strong hashing, inadequate salt or iteration counts can enable offline brute force attacks.

**Session Management Flaws:** Sessions that do not expire, use predictable identifiers, or are not invalidated on logout enable session hijacking and account takeover. Sessions transmitted over unencrypted connections are vulnerable to interception.

**Insecure Password Recovery:** Password reset mechanisms that do not verify user identity, send passwords via email, or use predictable reset tokens enable account takeover. An attacker who can reset a victim's password gains full account access.

**Lack of Multi-Factor Authentication:** Applications that rely solely on passwords for authentication are vulnerable to phishing, credential stuffing, and brute force attacks. MFA significantly raises the bar for attackers.

**Insufficient Authentication Logging:** Applications that do not log authentication events cannot detect account compromise or brute force attacks. Logging should capture successful and failed authentication attempts, including timestamp, username, and source IP.

### Authorization Risks

**Broken Access Control:** The most common authorization vulnerability is the failure to enforce access control checks. For example, an application might check authorization when displaying a resource but fail to check when the user attempts to modify it. An attacker can bypass the UI and directly request the modification endpoint.

**Privilege Escalation:** Vulnerabilities that allow a user to gain higher privileges than intended. For example, a user might modify a parameter in a request to change their role from "User" to "Administrator," or an application might fail to validate that a user has permission to perform an action on behalf of another user.

**Insecure Direct Object References (IDOR):** An application exposes object identifiers (such as user IDs or document IDs) in URLs or API parameters without proper authorization checks. An attacker can enumerate identifiers and access resources belonging to other users.

**Path Traversal and Directory Traversal:** An application fails to validate file paths, allowing an attacker to access files outside the intended directory. For example, a request for `/documents/../../etc/passwd` might return the system password file.

**Horizontal Privilege Escalation:** A user gains access to resources belonging to other users at the same privilege level. For example, a user might access another user's profile or financial records.

**Vertical Privilege Escalation:** A user gains access to resources or operations restricted to higher privilege levels. For example, a regular user might access administrative functions.

**Insecure Authorization Logic:** Authorization decisions based on user input, client-side checks only, or flawed logic. For example, an application might trust a user-supplied role parameter instead of retrieving the role from a secure session.

## Developer Lens

Developers must implement authentication and authorization correctly to prevent unauthorized access. This section provides practical guidance for secure implementation.

### Authentication Implementation

**Password Hashing:** Never store passwords in plaintext. Use a modern password hashing algorithm such as bcrypt, scrypt, or Argon2. These algorithms are intentionally slow, making brute force attacks computationally expensive. Use a unique salt for each password (modern algorithms handle this automatically).

```python
# Secure password hashing with bcrypt
import bcrypt

# During registration
password = "user_provided_password"
hashed = bcrypt.hashpw(password.encode('utf-8'), bcrypt.gensalt(rounds=12))
# Store hashed in database

# During login
provided_password = "user_provided_password"
if bcrypt.checkpw(provided_password.encode('utf-8'), stored_hashed):
    # Password matches, authenticate user
    pass
else:
    # Password does not match, reject login
    pass
```

**Password Policy:** Enforce minimum length (at least 12 characters) and check against common password lists. NIST SP 800-63B recommends prioritizing length and dictionary checking over complexity rules (uppercase, lowercase, numbers, symbols). Overly complex requirements encourage weak patterns such as predictable substitutions ("P@ssw0rd"). Consider using a password strength meter to guide users toward stronger passwords.

**Session Management:** Use secure, randomly generated session identifiers. Set appropriate expiration times (shorter for sensitive operations, longer for less sensitive). Invalidate sessions on logout. Use secure cookie flags: `HttpOnly` (prevents JavaScript access), `Secure` (HTTPS only), and `SameSite` (prevents CSRF and cross-site session fixation).

```javascript
// Setting a secure session cookie in Node.js/Express
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,        // HTTPS only
    httpOnly: true,      // No JavaScript access
    sameSite: 'strict',  // CSRF protection and session fixation prevention
    maxAge: 3600000      // 1 hour expiration
  }
}));
```

**Session Storage and Fixation Prevention:** Store sessions in a secure backend (Redis, database) rather than relying on client-side cookies alone. Regenerate session identifiers after successful authentication to prevent session fixation attacks, where an attacker forces a user to use a known session ID. Implement concurrent session limits to detect account takeover: if a user logs in from a new location, invalidate previous sessions or require re-authentication.

**Multi-Factor Authentication:** Implement MFA for sensitive operations or user accounts. Support multiple MFA methods (TOTP, push notifications, hardware tokens) to accommodate different user preferences and security requirements. Prioritize phishing-resistant methods such as FIDO2/WebAuthn hardware keys over SMS or email-based factors. Provide backup codes for account recovery if the primary MFA method is unavailable.

**Token-Based Authentication:** If using tokens, ensure they are signed with a strong algorithm. For distributed systems (microservices, APIs), use asymmetric algorithms (RS256 or ES256) where the authentication service signs tokens with a private key and applications validate with the public key. For single-service scenarios, HS256 with a strong shared secret is acceptable, but requires careful key distribution. Include an expiration time (`exp` claim) and validate it on every request. Use short-lived access tokens (15 minutes to 1 hour) and refresh tokens (days to weeks) to limit the window of compromise.

```javascript
// Validating a JWT token in Node.js
const jwt = require('jsonwebtoken');

function verifyToken(token) {
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      algorithms: ['RS256'],  // Asymmetric signature
      issuer: 'https://auth.example.com',
      audience: 'api.example.com'
    });
    return decoded;
  } catch (error) {
    // Token is invalid, expired, or tampered with
    return null;
  }
}
```

**Token Revocation and Expiration:** Short-lived access tokens (15 minutes to 1 hour) are the most practical approach to token revocation: when a token expires, the user must use a refresh token to obtain a new one. If immediate revocation is required (e.g., after logout or permission change), maintain a revocation list (blacklist) of invalidated tokens, but cache it locally to avoid querying on every request. Alternatively, implement token binding to tie tokens to specific clients or devices, preventing stolen tokens from being immediately useful to attackers.

**Password Reset:** Implement secure password reset mechanisms. Send a time-limited reset token (valid for 15-30 minutes) via email. Require the user to verify their identity (answer security questions, confirm email address) before allowing a reset. Never send passwords via email or display them in the UI. Invalidate reset tokens after single use.

**Authentication Logging:** Log all authentication events, including successful and failed login attempts, password changes, MFA enrollment, and session creation/termination. Include timestamp, username, source IP, and user agent. Monitor logs for suspicious patterns such as multiple failed attempts from the same IP, unusual login times, or logins from geographically impossible locations.

### Authorization Implementation

**Enforce Authorization Checks:** Check authorization before performing any sensitive operation, not just when displaying resources. Authorization checks should occur in the application logic, not just in the UI.

```python
# Flask example: Enforce authorization before modifying a document
from flask import Flask, request, abort
from functools import wraps

app = Flask(__name__)

def require_permission(permission):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            user = get_current_user()
            if not user or not user.has_permission(permission):
                abort(403)  # Forbidden
            return f(*args, **kwargs)
        return decorated_function
    return decorator

@app.route('/documents/<doc_id>', methods=['PUT'])
@require_permission('edit_documents')
def update_document(doc_id):
    # Authorization check passed, proceed with update
    document = Document.query.get(doc_id)
    if not document:
        abort(404)
    # Verify user owns or has access to this document
    if document.owner_id != get_current_user().id:
        abort(403)
    # Update document
    return update_logic(document)
```

**Principle of Least Privilege:** Grant users only the permissions they need. Regularly review and revoke unnecessary permissions. Use role-based access control to simplify permission management, but ensure roles are designed with PoLP in mind. Implement time-bound permissions for temporary access elevation.

**Avoid Client-Side Authorization:** Never rely on client-side checks (JavaScript, hidden form fields) for authorization. An attacker can bypass client-side checks by modifying the request directly. All authorization decisions must be made on the server.

**Validate User Input in Authorization Decisions:** Do not trust user-supplied parameters when making authorization decisions. For example, do not trust a user-supplied role parameter; retrieve the user's actual role from the session or token.

```javascript
// INSECURE: Trusting user-supplied role
app.post('/api/admin/users', (req, res) => {
  const role = req.body.role;  // User-supplied!
  if (role === 'admin') {
    // Perform admin action
  }
});

// SECURE: Retrieving role from session
app.post('/api/admin/users', (req, res) => {
  const user = req.session.user;
  if (user.role !== 'admin') {
    return res.status(403).send('Forbidden');
  }
  // Perform admin action
});
```

**Use Parameterized Queries:** When authorization logic involves database queries, use parameterized queries to prevent SQL injection, which could bypass authorization checks.

Penetration testers and security researchers assess authentication and authorization controls to identify vulnerabilities that could enable unauthorized access.

### Authentication Testing

**Credential Stuffing:** Test whether the application is vulnerable to credential stuffing attacks, where attackers use credentials compromised in other breaches. Attempt login with common username/password combinations. Check whether the application implements rate limiting or account lockout after multiple failed attempts.

**Brute Force Attacks:** Attempt to brute force user passwords. Test whether the application implements progressive delays, account lockout, or CAPTCHA challenges after multiple failed attempts. Measure how quickly an attacker could compromise an account.

**Password Policy Assessment:** Review the password policy for weaknesses. Test whether the application accepts weak passwords (short length, no complexity). Check whether the application prevents password reuse or enforces password expiration.

**Session Fixation:** Attempt to force a user to use a known session identifier. For example, set a session cookie before the user logs in, then verify whether the application uses the same session after login. If so, an attacker could hijack the session.

**Session Hijacking:** Attempt to steal or predict session identifiers. Analyze session tokens for patterns or predictability. Test whether sessions are transmitted over HTTPS. Attempt to use a session from another user.

**Password Reset Vulnerabilities:** Test the password reset mechanism for flaws. Attempt to reset another user's password. Test whether reset tokens are time-limited and single-use. Attempt to predict reset tokens. Test whether the application verifies user identity before allowing a reset.

**MFA Bypass:** Test whether MFA can be bypassed. Attempt to access the account without providing the second factor. Test whether backup codes are properly protected. Attempt to brute force TOTP codes (though this is difficult with proper implementation).

### Authorization Testing

**Insecure Direct Object References (IDOR):** Attempt to access resources belonging to other users by modifying object identifiers in URLs or API parameters. For example, change `/api/users/123/profile` to `/api/users/124/profile` and check whether you can access another user's profile.

**Privilege Escalation:** Attempt to gain higher privileges than intended. Test whether you can modify your role or permissions. Attempt to perform administrative actions as a regular user. Test whether you can perform actions on behalf of other users.

**Path Traversal:** Attempt to access files outside the intended directory. Test file download and upload functionality for path traversal vulnerabilities. For example, attempt to download `/documents/../../etc/passwd`.

**Horizontal Privilege Escalation:** Attempt to access resources belonging to other users at the same privilege level. For example, access another user's documents, messages, or financial records.

**Vertical Privilege Escalation:** Attempt to access resources or operations restricted to higher privilege levels. For example, access administrative functions as a regular user.

**Authorization Logic Flaws:** Analyze authorization logic for flaws. Test edge cases and boundary conditions. For example, test whether authorization checks are applied consistently across all endpoints. Test whether authorization is enforced for all HTTP methods (GET, POST, PUT, DELETE).

**Timing Attacks:** Analyze response times to infer information about authorization decisions. For example, if the application takes longer to respond when denying access to a resource that exists versus a resource that does not exist, an attacker could enumerate resources.

## Common Findings

### Finding 1: Weak Password Policy

**Description:** The application accepts passwords shorter than 12 characters and does not enforce complexity requirements.

**Risk:** Users choose weak passwords that are vulnerable to brute force and dictionary attacks.

**Example:** A user sets the password "password123" (11 characters, predictable pattern). An attacker with a dictionary of common passwords compromises the account in minutes.

**Remediation:** Enforce a minimum password length of 12 characters. Avoid overly complex requirements; instead, use a password strength meter and check against common password lists. Consider using passphrases (multiple words) instead of complex passwords.

### Finding 2: Insecure Direct Object References (IDOR)

**Description:** The application does not verify that a user has permission to access a resource before returning it.

**Risk:** Users can access resources belonging to other users by modifying object identifiers.

**Example:** A user accesses their own profile at `/api/users/123/profile`. They modify the URL to `/api/users/124/profile` and receive another user's profile data, including email address and phone number.

**Remediation:** Verify authorization before returning any resource. Check that the authenticated user has permission to access the requested resource. Use indirect references (such as UUIDs) instead of sequential IDs to make enumeration more difficult.

## Secure Design Guidance

### Authentication Design Principles

**Design for Defense in Depth:** Authentication should not rely on a single mechanism. Combine multiple factors, implement rate limiting, use secure session management, and maintain comprehensive logging. If one control fails, others remain in place.

**Separate Authentication from Authorization:** Design systems where authentication and authorization are independent concerns. A user may be successfully authenticated but still lack authorization for a specific resource. This separation enables flexible policy changes without modifying authentication logic.

**Minimize Credential Exposure:** Reduce the number of times credentials are transmitted and stored. Use token-based authentication to avoid repeatedly sending passwords. Implement credential rotation policies. Never log or display credentials in error messages or debug output.

**Design for Token Lifecycle Management:** If using tokens, design explicit lifecycle policies: short expiration times for access tokens (15 minutes to 1 hour), longer expiration for refresh tokens (days to weeks), and mechanisms to revoke tokens before expiration. Consider token binding to prevent token theft from being immediately useful to attackers.

**Implement Graceful Degradation:** Design authentication systems that continue functioning during partial failures. For example, if a multi-factor authentication service is temporarily unavailable, the system should either queue the authentication request or fall back to a secure alternative rather than denying all access.

**Design Audit Trails from the Start:** Authentication and authorization decisions should be logged as a core architectural requirement, not an afterthought. Design systems to capture: who attempted access, what they attempted to access, when the attempt occurred, from where, and what the decision was. Ensure logs are immutable and protected from tampering.

### Authorization Design Principles

**Apply Principle of Least Privilege at Design Time:** Rather than granting broad permissions and restricting later, design systems where users receive minimal permissions by default. Require explicit approval for elevated permissions. Regularly audit and revoke unnecessary permissions.

**Design Role Hierarchies Carefully:** If using role-based access control, design role hierarchies that reflect organizational structure and job functions. Avoid creating overly broad roles that combine unrelated permissions. Document the purpose and permissions of each role.

**Implement Attribute-Based Access Control for Complex Scenarios:** For applications with complex authorization requirements (healthcare, finance, multi-tenant systems), consider attribute-based access control. Design policies that evaluate user attributes (department, clearance level), resource attributes (classification, owner), and environmental attributes (time of day, network location).

**Separate Policy from Enforcement:** Design systems where authorization policies are defined separately from the code that enforces them. This enables policy changes without code deployment and allows non-technical stakeholders to review and approve policies.

**Design for Revocation:** Authorization systems should support immediate revocation of access. Design mechanisms to revoke permissions without requiring user re-authentication. For token-based systems, implement token revocation lists or short expiration times to ensure revoked permissions take effect quickly.

**Design Consistent Authorization Across Interfaces:** Authorization logic should be consistent whether users access the application via web UI, API, mobile app, or batch processes. Avoid implementing authorization only in the UI; enforce it at the business logic layer.

### Multi-Tenant Authorization Design

For applications serving multiple organizations or customers, authorization design must prevent cross-tenant access:

**Tenant Isolation:** Every data query and operation should be scoped to the authenticated user's tenant. Use tenant identifiers in database queries, not just in the application logic. Verify tenant membership before returning any data.

```python
# Secure multi-tenant query
def get_user_documents(user_id, tenant_id):
    # Verify user belongs to tenant
    user = User.query.filter_by(id=user_id, tenant_id=tenant_id).first()
    if not user:
        raise Unauthorized()
    
    # Query documents scoped to tenant
    documents = Document.query.filter_by(
        owner_id=user_id,
        tenant_id=tenant_id  # Critical: scope to tenant
    ).all()
    return documents
```

**Tenant-Aware Tokens:** If using tokens, include the tenant identifier in the token claims. Verify on every request that the token's tenant matches the requested resource's tenant.

**Separate Tenant Credentials:** Each tenant should have separate credentials and secrets. Avoid sharing encryption keys, API keys, or signing keys across tenants.

## Key Takeaways

1. **Authentication and Authorization Are Distinct but Inseparable:** Authentication verifies identity; authorization determines permissions. Both must be implemented correctly. A successfully authenticated user still requires authorization checks to prevent unauthorized access.

2. **Multi-Factor Authentication Significantly Raises the Security Bar:** Single-factor authentication (password only) is vulnerable to phishing, credential stuffing, and brute force attacks. MFA, combining something you know with something you have or are, makes account compromise substantially more difficult.

3. **Authorization Checks Must Occur on the Server, Not the Client:** Client-side authorization checks (JavaScript, hidden fields) can be bypassed. All authorization decisions must be made on the server, after verifying the user's identity and permissions from a trusted source.

4. **Principle of Least Privilege Reduces the Impact of Compromise:** Users should have only the minimum permissions necessary for their role. If an account is compromised, the attacker's access is limited to what that user could legitimately access. Regularly audit and revoke unnecessary permissions.

5. **Insecure Direct Object References (IDOR) Are Among the Most Common Authorization Vulnerabilities:** Applications frequently expose object identifiers (user IDs, document IDs) without verifying authorization. Always verify that the authenticated user has permission to access the requested resource before returning it.

6. **Session Management Requires Careful Attention to Expiration, Invalidation, and Secure Transmission:** Sessions should expire after a period of inactivity, be invalidated on logout, use secure cookie flags (HttpOnly, Secure, SameSite), and be transmitted only over HTTPS. Predictable or reused session identifiers enable session hijacking.

7. **Password Hashing Must Use Modern Algorithms with Appropriate Iteration Counts:** Never store passwords in plaintext or with weak algorithms (MD5, SHA1). Use bcrypt, scrypt, or Argon2 with appropriate iteration counts (bcrypt rounds ≥ 12). These algorithms are intentionally slow, making offline brute force attacks computationally expensive.

8. **Comprehensive Audit Logging Enables Detection and Investigation of Unauthorized Access:** Log all authentication and authorization events, including successful and failed attempts, with timestamp, user, action, resource, and decision. Monitor logs for suspicious patterns such as multiple failed attempts or unusual access times.

9. **Token-Based Authentication Requires Careful Management of Expiration, Refresh, and Revocation:** Access tokens should be short-lived (minutes to hours); refresh tokens longer-lived (days to weeks). Tokens must be signed with strong algorithms and validated on every request. Implement mechanisms to revoke tokens before expiration.

10. **Multi-Tenant Systems Require Explicit Tenant Isolation in Authorization Logic:** Every query and operation must be scoped to the authenticated user's tenant. ## Pentest Lens

Penetration testers and security researchers assess authentication and authorization controls to identify vulnerabilities that could enable unauthorized access.

### Authentication Testing

**Credential Stuffing:** Test whether the application is vulnerable to credential stuffing attacks, where attackers use credentials compromised in other breaches. Attempt login with common username/password combinations. Check whether the application implements rate limiting, account lockout, or CAPTCHA challenges after multiple failed attempts. Measure how quickly an attacker could compromise an account. Test for account enumeration: do error messages or response times reveal whether a username exists?

**Brute Force Attacks:** Attempt to brute force user passwords. Test whether the application implements progressive delays, account lockout, or CAPTCHA challenges after multiple failed attempts. Measure how quickly an attacker could compromise an account. Test whether lockout mechanisms can be bypassed (e.g., by resetting the counter or using different usernames).

**Password Policy Assessment:** Review the password policy for weaknesses. Test whether the application accepts weak passwords (short length, no complexity). Check whether the application prevents password reuse or enforces password expiration. Verify that the application checks against common password lists.

**Session Fixation:** Attempt to force a user to use a known session identifier. For example, set a session cookie before the user logs in, then verify whether the application uses the same session after login. If so, an attacker could hijack the session. Verify that the application regenerates session identifiers after successful authentication.

**Session Hijacking:** Attempt to steal or predict session identifiers. Analyze session tokens for patterns or predictability. Test whether sessions are transmitted over HTTPS. Attempt to use a session from another user. Test whether the application validates the user agent or IP address associated with a session (to detect session theft from different locations).

**Concurrent Session Detection:** Test whether the application detects and prevents concurrent sessions from different locations. Authenticate as the same user from two different IP addresses or browsers simultaneously. Verify whether the application invalidates the previous session or alerts the user to the new login.

**Password Reset Vulnerabilities:** Test the password reset mechanism for flaws. Attempt to reset another user's password. Test whether reset tokens are time-limited and single-use. Attempt to predict reset tokens. Test whether the application verifies user identity before allowing a reset. Check for account enumeration in the reset flow (does the application reveal whether an email is registered?).

**MFA Bypass:** Test whether MFA can be bypassed. Attempt to access the account without providing the second factor. Test whether backup codes are properly protected and single-use. Attempt to brute force TOTP codes (though this is difficult with proper implementation). Test whether the application allows fallback to weaker authentication methods if MFA fails. Test for race conditions: can you complete login before MFA validation completes?

**Passwordless Authentication Testing:** If the application supports passwordless methods (WebAuthn/FIDO2), test whether the implementation correctly validates attestation and assertion objects. Verify that the application binds credentials to specific users and devices. Test whether the application prevents credential cloning or reuse.
